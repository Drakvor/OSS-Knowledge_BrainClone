## 📊 **SECTION I: 프로세스 모델링 (56문항)**

### **A. 분석 영역 (15문항)**

#### **현행 시스템 분석 (4문항)**
| 기법 | 내용 |
|------|------|
| **PIECES** | **P**erformance, **I**nformation, **E**conomy, **C**ontrol, **E**fficiency, **S**ervice |
| **FAST** | Facilitated Application Specification Technique (워크숍 기반) |
| **역공학** | 소스코드/DB 구조 분석 → 문서화 |

**문제점 구분:** 기능적 vs 비기능적 (성능 = 비기능적)

#### **요구사항 정의 (6문항)**
| 도출 기법 | 특징 |
|-----------|------|
| **인터뷰** | 일대일/그룹 면담 |
| **브레인스토밍** | 자유로운 아이디어 수집 |
| **프로토타이핑** | 시제품 피드백 |
| **워크숍** | Joint Application Development |
| **현장조사** | 실제 업무 환경 관찰 (시간/비용 多) |

**MoSCoW 우선순위:** Must have > Should have > Could have > Won't have  
**SMART 원칙:** Specific, Measurable, Achievable, Relevant, Time-bound  
**요구사항 분류:** 기능적 (서비스/기능) vs 비기능적 (성능/보안/품질)

#### **화면/인터페이스 정의 (5문항)**
**UI 설계:** 와이어프레임 (구조+기능 중심, 상세 컬러/폰트 제외) → 스토리보드 → IA  
**제스탈트 법칙:** 근접성, **유사성** (유사 요소 그룹화), 연속성, 폐쇄성  
**햄버거 메뉴:** 모바일/작은 화면용

### **B. 설계 영역 (41문항)**

#### **화면 설계 (8문항)**
**반응형 웹:** Mobile First 접근법 (핵심 기능 집중, 점진적 개선)  
**IA 설계:** 콘텐츠 구조화, 사용자 멘탈 모델, 네비게이션 (DB 스키마 제외)

#### **인터페이스 설계 (12문항)**
| HTTP 상태 코드 | 의미 |
|----------------|------|
| **200** | OK (성공) |
| **201** | Created (리소스 생성) |
| **400** | Bad Request (잘못된 요청) |
| **401** | Unauthorized (권한 없음) |
| **500** | Internal Server Error (**서버** 오류) |

**API 설계:**
- **OpenAPI 명세:** servers 섹션 필수 (API 서버 URL 정보)
- **gRPC:** Protocol Buffers 사용 (JSON 아님), HTTP/2, 스트리밍
- **API 버전 관리:** URL 경로 포함 방식 (예: `/v1/users`)

**마이크로서비스:**
- **계약 테스트:** Pact 도구 사용
- **API Gateway:** 중앙 라우팅, 인증/로깅/모니터링, 클라이언트로부터 서비스 위치/프로토콜 숨김

#### **비즈니스 로직 설계 (8문항)**
**SOLID 원칙 (암기 필수):**
| 원칙 | 내용 |
|------|------|
| **SRP** | 단일 책임: 하나의 클래스는 하나의 책임 |
| **OCP** | 개방-폐쇄: **확장에 열려있고 수정에 닫혀있음** |
| **LSP** | 리스코프 치환: 서브타입은 기반타입 대체 가능 |
| **ISP** | 인터페이스 분리: 구체적인 여러 인터페이스 > 하나의 인터페이스 |
| **DIP** | 의존성 역전: 변화하기 쉬운 것에 의존 금지 |

**GoF 디자인 패턴:**
- **생성:** Singleton, Factory Method, Abstract Factory, Builder, Prototype
- **구조:** Adapter, Bridge, Composite, Decorator, Facade, Flyweight, Proxy
- **행위:** **Observer**(일대다 종속, 상태변경 알림), Strategy, Command, State

**아키텍처 패턴:**
- **MVC:** Model-View-Controller (관심사 분리, 결합도↓, 재사용성↑)
- **MVVM:** Model-View-ViewModel (데이터 바인딩)

**함수형 프로그래밍:** 불변성, 순수함수, 고차함수, 함수합성 (객체 상태 변경 지양)

#### **테스트 케이스 설계 (11문항)**
**TDD 사이클:** 테스트 → 구현 → 리팩토링 (Red → Green → Refactor)  
**순환 복잡도:** 결정점 수 + 1

**테스트 레벨:**
1. **단위 테스트:** 개별 모듈/함수
2. **통합 테스트:** 모듈 간 연동
3. **시스템 테스트:** 전체 시스템 요구사항
4. **인수 테스트:** 최종 사용자 승인

**테스트 기법:**
| 구분 | 기법 | 특징 |
|------|------|------|
| **명세기반 (Black-box)** | 동등분할(EP) | 입력 도메인을 대표값 그룹으로 나눔 |
| | **경계값분석(BVA)** | 최솟값, 최댓값, 경계 전후값 테스트 |
| | 원인-결과 그래프 | 입출력 관계 그래프화 |
| | 상태 전이 | 시스템 상태 변화 기반 |
| **구조기반 (White-box)** | 구문 커버리지 | 모든 문장 실행 |
| | 분기 커버리지 | 모든 분기문 실행 |
| | 조건 커버리지 | 모든 조건의 참/거짓 |
| **경험기반** | 오류 추정 | 과거 경험 기반 |
| | 탐색적 테스트 | 자유로운 탐색 |

**통합 테스트:**
- **Big Bang:** 모든 모듈 한번에 (결함 위치 파악 어려움)
- **Test Double:** Mock, Stub (외부 시스템 대체)

#### **신기술/아키텍처 설계 (2문항)**
**Kubernetes 구성요소:** Pod, Service, Ingress, ConfigMap (Repository 제외)  
**이벤트 소싱:** 감사추적 제공, 시점별 재구성 (쿼리 성능은 느림)

## 💾 **SECTION II: 데이터 모델링 (44문항)**

### **A. 데이터 표준화 (5문항)**
**구성 요소:** 표준단어 → 표준용어 → 표준도메인 → 코드  
**도메인:** 데이터 타입, 길이, 허용값 범위 표준화 (예: 등록금_N8 = 0~99,999,999)

### **B. 데이터 모델링 (39문항)**

#### **논리 데이터 모델링 (24문항)**

**데이터 모델링 단계:**
| 단계 | 특징 | 산출물 |
|------|------|--------|
| **개념적** | 추상화 수준 높음, 업무 중심, 포괄적 | 고수준 ERD |
| **논리적** | Key, 속성, 관계 정확히 정의, 재사용성 높음 | 상세 ERD, 정규화 |
| **물리적** | DBMS 이식, 성능/물리적 특성 고려 | DDL, 인덱스 |

**정규화 (암기 필수):**
| 정규형 | 조건 | 제거 대상 |
|--------|------|-----------|
| **1NF** | 원자값, 반복그룹 제거 | 복수값 속성 |
| **2NF** | 1NF + 부분 함수 종속 제거 | 복합키 일부 종속 |
| **3NF** | 2NF + 이행적 함수 종속 제거 | 비키 속성 간 종속 |
| **BCNF** | 모든 결정자가 후보키 | 결정자 종속 |
| **4NF** | 다치 종속 제거 | 독립적 다중값 |
| **5NF** | 조인 종속 제거 | 조인 종속성 |

**데이터 이상 (Anomaly):**
- **갱신 이상:** 중복 데이터 불일치
- **삽입 이상:** 불필요한 중복 데이터 삽입 필요
- **삭제 이상:** 의도치 않은 정보 손실

**ERD 구성:**
- **엔티티:** 사각형 (관리 대상 그룹)
- **관계:** 마름모/선 (1:1, 1:N, N:M)
- **속성:** 타원 (Primary Key 밑줄, Foreign Key 표시)

**개념→논리 변환 시 추가:** 상세 속성 및 데이터 타입

#### **물리 데이터 모델링 (15문항)**

**인덱스 설계:**
- **B-Tree:** 일반적 검색 (기본)
- **함수기반:** UPPER(), LOWER() 등 함수 적용 검색
- **비트맵:** 카디널리티 낮은 컬럼

**파티셔닝:**
| 유형 | 적합한 데이터 |
|------|---------------|
| **Range** | 날짜, 순차적 데이터 |
| **List** | 지역코드, 고정 분류값 |
| **Hash** | 균등 분산 필요 시 |

**반정규화 기법:**
- **테이블 통합:** 1:1, 1:M 관계 병합
- **중복 컬럼 추가:** 조인 감소 목적
- **통계 테이블:** SUM, AVG 미리 계산
- **수직분할:** 컬럼 단위 분할
- **수평분할:** 값 기준 테이블 분할

**테이블스페이스:** 로그파일 ↔ 데이터파일 분리 (I/O 분산, 성능 향상)

## 📐 **규모 산정 (Function Point)**

### **FP 5대 기능 유형**
| 유형 | 설명 | 예시 |
|------|------|------|
| **EI** | External Input | 회원가입 폼, 상품 주문 |
| **EO** | External Output | 보고서 출력, 확인 메시지 |
| **EQ** | External Inquiry | 상품 검색, 재고 조회 |
| **ILF** | Internal Logical File | 회원정보 DB, 주문내역 DB |
| **EIF** | External Interface File | 외부 배송사 데이터, 공공 API |

**산정 공식:**
```
개발비용 = (AFP × FP당 단가 × 난이도 보정계수) + 직접경비 + 이윤
생산성 = 총 FP / 총 소요 인월
```

**복잡도:** Low/Average/High → 점수 차등 적용  
**보정계수(VAF):** 0.65~1.35 (14개 시스템 특성 평가)

## 🏗️ **데이터베이스 핵심**

### **관계형 모델 기본**
**3단계 구조:** 외부 스키마 (사용자뷰) ↔ 개념 스키마 (통합논리) ↔ 내부 스키마 (물리저장)  
**데이터 독립성:** 논리적 + 물리적 독립성

**기본 용어:**
- **릴레이션:** 테이블 구조
- **스키마:** 논리적 구조 정의
- **인스턴스:** 특정 시점 데이터 집합
- **관계:** 테이블 간 논리적 연결

### **키(Key) 설계**
**기본키(PK):** 유일 식별자, NULL 불가  
**외래키(FK):** 다른 테이블 기본키 참조  
**대체키:** 비즈니스키 변경 시 영향 최소화, 보안성 강화

## 🚀 **최신 기술 및 아키텍처**

### **마이크로서비스 아키텍처(MSA)**
**핵심 패턴:**
| 패턴 | 설명 |
|------|------|
| **Saga** | 분산 트랜잭션: Choreography (자율) vs Orchestration (중앙집중) |
| **CQRS** | 명령(Command)과 쿼리(Query) 분리 |
| **Event Sourcing** | 이벤트 기반 상태 관리 (감사추적 Good, 쿼리 성능 Bad) |
| **API Gateway** | 중앙 라우팅, 공통 기능, 서비스 위치 숨김 |

### **클라우드 네이티브**
**특성:** 민첩성(Agility) + 탄력성(Elasticity) + 비용효율성  
**핵심요소:** 마이크로서비스 + 컨테이너 + DevOps + 자동화

**Kubernetes 구성요소:**
- **Pod:** 가장 작은 배포 단위
- **Service:** 네트워크 서비스 추상화
- **Ingress:** 외부 트래픽 라우팅
- **ConfigMap:** 설정 데이터 관리
- ❌ **Repository:** Git 버전관리 (Kubernetes 아님)

**카오스 엔지니어링:** 장애 상황 임의 조성으로 안정성 테스트

- **도메인:** 데이터 타입/길이/범위 표준화
- **정규화:** 데이터 중복 제거, 이상 방지
- **반정규화:** 성능 향상을 위한 의도적 중복 허용
- **결정테이블:** 복합 조건 조합 분석
- **현장조사:** 실제 업무 환경 직접 관찰
- **순환복잡도:** 코드 복잡도 측정 (결정점 수 + 1)

## 🔍 **자주 출제되는 계산 문제**

### **정규화 테이블 분리**
**2NF 적용 시:** 부분 함수 종속 제거 → 별도 테이블 생성  
**예시:** 주문상세(주문번호+상품코드) → 주문 + 상품 + 주문상세 (3개 테이블)

### **순환 복잡도 계산**
```java
public String getGrade(int score) {
    if (score >= 90) return "A";        // 결정점 1
    else if (score >= 80) return "B";   // 결정점 2  
    else if (score >= 70) return "C";   // 결정점 3
    else return "F";
}
// 순환 복잡도 = 3 + 1 = 4
```

### **할인 계산 로직**
1. 기본 할인 적용
2. 회원 할인 추가 적용  
3. 쿠폰 할인 최종 적용 (정액)

## 🚨 **시험 팁**

### **함정 주의사항**
- **500 에러 = 서버 오류** (클라이언트 오류 아님)
- **gRPC = Protocol Buffers** (JSON 아님)
- **이벤트 소싱 = 쿼리 성능 느림** (장점 아님)
- **함수형 프로그래밍 ≠ 객체 상태 변경**
- **Repository ≠ Kubernetes 구성요소**

### **단답형 키워드**
- 모바일 환경 → **모바일** (또는 작은 화면)
- 중앙 라우팅 패턴 → **API Gateway**
- 계약 테스트 도구 → **Pact**
- 경계값 분석 → **BVA**
- 애자일 개발 주기 → **Sprint**
- 실시간 응답 → **Real Time**

### **계산 문제 공식**
- **순환 복잡도 = 결정점 수 + 1**
- **FP 생산성 = 총 FP / 총 소요 인월**
- **2NF 분리:** 부분 종속 속성별로 별도 테이블
